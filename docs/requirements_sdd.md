# Software Design & Requirements Document (SDD): Smart Solar EV Charging

This canonical requirements register specifies the functional, non-functional, and security constraints for the Smart Solar EV Charging application.

---

## 1. Functional Requirements (FR)

### FR-1: Real-Time Multi-Tab Dashboard (UI/UX)
*   **Description:** The front-end must render a premium, responsive (mobile/desktop), glassmorphic dashboard with multiple tabs.
*   **Views Required:**
    1.  **Real-Time Status Tab:** Renders an interactive SVG power-flow diagram showing live solar generation, household load, net grid import/export, and vehicle charging power.
    2.  **Historical Logs Tab:** Displays past charging session metrics, total solar energy diverted to the EV (kWh), and dynamic charge level history.
    3.  **Battery & System Settings Tab:** Exposes network configurations, voltage/phase inputs, and custom safety thresholds.
    4.  **Advanced Telemetry Charts Tab:** Plots raw solar output vs. vehicle current over time.

### FR-2: Dynamic Charging Control Loop
*   **Description:** The backend must execute a 10-second polling loop to adjust charging amperage based on smoothed active power.
*   **Loop Calculations (Three-Phase):**
    *   Amperage steps scale at **~690 Watts per Ampere** ($3 \times 230\text{V}$ AC).
    *   Minimum charging rate is **5A** ($3,450\text{W}$). Charging triggers only when smoothed export exceeds $3,450\text{W}$.
    *   Maximum charging rate is clamped at **16A** ($11,040\text{W}$).

### FR-3: Aggressive Cloud-Drop Hysteresis
*   **Description:** To maximize zero-grid-export efficiency, the system will respond aggressively to transient solar drops (passing clouds).
*   **Behavior:**
    *   The backend polls the inverter every 10 seconds.
    *   If net active power drops below 0W (indicating grid import), the system must **immediately (within 10 seconds) adjust the current down to 5A or completely pause/suspend charging**.
    *   No long delay windows or grid-import tolerance are permitted.

### FR-4: Manual Override Lockout
*   **Description:** The user can manually toggle "Manual Mode" or adjust charging current on a slider via the UI.
*   **Lockout Heuristic:**
    *   Toggling manual override suspends automated solar tracking immediately.
    *   The manual current setting is locked for a configurable duration (defaulting to **2 hours**).
    *   Once the lockout timer expires, the system automatically transitions back to smart solar tracking mode.

### FR-5: Tesla API Throttling & Smoothing
*   **Description:** To stay within Tesla API rate limits and avoid vehicle sleep disruption, the system must filter outbound commands.
*   **Throttling Rule:**
    *   While active, the backend polls the inverter every 10 seconds.
    *   The Tesla charge current is only modified if the target amperage changes by **$\ge 2\text{A}$** OR if the system needs to prevent grid import (immediate drop).

## 2. Non-Functional Requirements (NFR)

### NFR-1: Connectivity & Protocols
*   **Inverter Protocol:** Local UDP polling on port `8899` via the async Python `goodwe` library.
*   **Vehicle Protocol:** Open-source Python `teslapy` library communicating with the Tesla Owner API / Fleet API.
*   **Network:** The inverter must have a static local IP.

### NFR-2: Performance & Scalability
*   **Dashboard Latency:** Telemetry must stream from the Python backend to the UI with a latency of $< 1\text{ second}$ using WebSockets.
*   **Responsive Breakpoints:** Fully fluid layout support down to $320\text{px}$ (mobile) and up to $4\text{K}$ resolution.

### NFR-3: Security & Token Caching
*   **Token Security:** The `cache.json` token container generated by `teslapy` must be stored as an **AES-256 encrypted file** at rest.
*   **In-Memory Decryption:** Decrypted in-memory on startup using a secure environmental key (`VAULT_SECRET`).
*   **File Permissions:** Standard host file permissions locked to `600` (owner read/write only).

## 3. Definition of Ready (DoR) & Done (DoD)

### Definition of Ready (DoR)
*   [x] All core functional requirements are documented and approved by the user.
*   [x] Inverter and TeslaPy integration protocols are researched and locked down.
*   [x] Target development stack is confirmed (Python, FastAPI, SQLite, Vanilla JS).

### Definition of Done (DoD)
*   [ ] Backend code completes all core telemetry calculations with $100\%$ unit test coverage.
*   [ ] Front-end multi-tab dashboard displays fluid SVG animations and manual override options.
*   [ ] Security audit confirms AES-256 encryption at rest for TeslaPy credentials.
*   [ ] Entire system passes local integration tests without any unhandled exceptions.
